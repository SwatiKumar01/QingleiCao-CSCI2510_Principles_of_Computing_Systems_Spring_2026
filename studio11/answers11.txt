4. Original racy program timings (20,000,000 iterations):

Run 1: 0.300 sec
Run 2: 0.012 sec
Run 3: 0.012 sec

Average: 0.108 sec

The final value changed every run and was not 0, showing data corruption caused by race conditions.


5. Corrected program with lock/unlock every iteration:

Run 1: 0.762 sec
Run 2: 0.490 sec
Run 3: 0.522 sec

Average: 0.591 sec

This took longer because each thread performs millions of lock and unlock operations. The mutex adds synchronization overhead.


6. Program with one lock/unlock per thread:

Run 1: 0.448 sec
Run 2: 0.096 sec
Run 3: 0.096 sec

Average: 0.213 sec

This version was much faster than locking every iteration because only two total mutex lock/unlock operations were used.


7. Why are the results different?

Per-iteration locking causes much higher overhead because the mutex is locked and unlocked millions of times. Locking once per thread greatly reduces synchronization cost.


8. When is each strategy appropriate?

Per-iteration locking is better when multiple threads must safely update shared data many times, such as counters, queues, or bank account balances.

Per-thread locking is better when one thread needs exclusive access to complete a large section of work before another thread begins.
